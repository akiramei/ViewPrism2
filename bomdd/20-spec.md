## 1. 概要と用語

ViewPrism2 は Windows 向けデスクトップ画像管理アプリケーション(.NET 10 / Avalonia UI 12 / SQLite)。
中核コンセプトは「タグ × 仮想ビュー」: 物理フォルダ階層に依存せず、タグ付けした画像を
ユーザー定義の仮想階層(ビュー)でフォルダのように辿る。

| 用語 | 定義 |
|---|---|
| 同期フォルダ | ユーザーが登録する物理フォルダ。スキャン対象の根。原典では「コレクション」とも呼ぶ |
| 画像(image) | 同期フォルダ配下で発見された画像ファイルの DB レコード。実体はファイルシステム上に残る(取り込みコピーはしない) |
| ステータス | 画像の状態: normal(正常)/missing(物理消失)/deleted(論理削除)/pending(再リンク候補確定待ち) |
| タグ | 画像に付与する分類子。simple(値なし)/textual(文字列値)/numeric(数値)の 3 種 |
| ビュー | 絞り込み条件・ソート・表示列・タグ階層を束ねた保存可能な仮想フォルダ定義 |
| タグ階層(hierarchy) | ビュー内に定義する木構造。各ノードがタグ(+値条件)を参照する |
| NodeGraph | タグ階層と実際のタグ付け状態から構築される、ユーザーが辿る展開済みノード木 |
| 再リンク | missing 画像と新発見ファイル(pending)を同一物として結合し、タグ等の関連を保持する操作 |
| 表示モード(ビューア) | ビューアの画像配置方式: normal(単一)/scroll(縦連続)/spread-right(右開き見開き)/spread-left(左開き見開き)(v2.0) |
| ページ送り | 見開きモードで現在位置を進める/戻す操作。ステップは 2(doublePage 既定)または 1(singlePage / SHIFT 併用)(v2.0) |
| pHash(知覚ハッシュ) | 画像の見た目を表す 64bit の指紋。DCT ベースで算出し、似た画像は近い値になる(v3.0) |
| ハミング距離 | 2 つの pHash の異なるビット数(0〜64)。小さいほど似ている。類似度%へ変換する(v3.0) |
| 類似検索 | 基準画像 1 枚に対し、pHash の近い画像を同一コレクション内から類似度順に集める操作(v3.0) |
| マージ | 重複画像を 1 枚に統合する論理操作。タグを集約しマージ元を deleted にする。物理ファイルは触れない(v3.0) |

## 2. 機能仕様

### 2.0 論理データモデル (REQ-001〜REQ-005)

**DB 基盤(REQ-003〜005 — v2.0 で本文へ明記。意味論は v1 から不変)**:
- DB ファイルは `%APPDATA%/ViewPrism2/data/viewprism.db` に配置する (REQ-005)
- 接続は journal_mode=WAL・foreign_keys=ON で開く(接続毎に PRAGMA を明示発行) (REQ-003)
- スキーマ変更はマイグレーション基盤で管理する: migrations テーブル(id TEXT PK, executed_at)に適用履歴を
  記録し、未適用分のみを id 順にトランザクション内で適用する。新規 DB は最新スキーマで作成して全
  マイグレーション id を適用済みとしてマークする (REQ-004)

エンティティと関係(物理スキーマは E-BOM/M-BOM で確定。カラム名は原典準拠):

- **sync_folders** 1—n **images**(フォルダ削除で画像も削除)
- **images** n—n **tags**(中間 **image_tags**、PK=(image_id, tag_id)、value 列に値)
- **tags** は自己参照 parent_id(単一親階層)。type 別設定として textual_tag_settings(predefined_values JSON 配列)、numeric_tag_settings(min/max/step/unit)
- **views** 1—n **view_conditions**(絞り込み条件)、1—n **view_tag_hierarchies**(階層ノード、自己参照 parent_id)
- **(v3.0)** **images** 1—1 **image_features**(pHash 特徴量。image_id PK、FK→CASCADE)、**images** n—n **image_similarity**(画像ペアの類似度キャッシュ。image_id1/image_id2 とも FK→CASCADE)。**列定義・索引・cache_key 生成規則は §2.10.3**。両テーブルはマイグレーション基盤(REQ-004)で追加(§2.10 REQ-063)。L2 スキーマ検査は v3.0 の 2 テーブルを含めて新規作成 DB と照合する
- ID は全エンティティ UUIDv4 小文字(REQ-001)。日時は ISO 8601 UTC 文字列 `YYYY-MM-DDTHH:mm:ss.fffZ`(REQ-002)
- 参照整合性は FK で固定: images.sync_folder_id → CASCADE / image_tags 両 FK → CASCADE /
  view_conditions.view_id → CASCADE, .tag_id → SET NULL / view_tag_hierarchies.view_id, .tag_id → CASCADE,
  .parent_id(自己参照)→ SET NULL / tags.parent_id → SET NULL
- 核/表面: core
- 受入観点: スキーマ検査(L2: PRAGMA/information クエリで FK・既定値・索引を照合)

### 2.1 同期フォルダとスキャン (REQ-010〜REQ-017)

**登録・管理 (REQ-010)**
- フィールド: name(表示名)、path(絶対パス)、is_active(既定 true)、include_subfolders(既定 true)、
  exclude_patterns(文字列配列、ファイル名の完全一致・大文字小文字無視で除外。例 `Thumbs.db`。glob/regex は対象外)、
  last_scan(NULL 初期)
- path はシステム内で一意(大文字小文字無視)。重複登録は明示エラーで拒否。name の重複は許可
- is_active=false のフォルダはスキャン・表示の対象外
- 削除時は配下 images が連鎖削除される(確認ダイアログ必須 — タグ関連も消えるため)

**スキャン (REQ-011〜015)**

> **実行順序(重要・ECO-001 で明確化)**: 以下 1〜6 は論理的記述順であり、実行順は **手順 4・5(missing 化・孤立 pending 削除)を手順 3(各ファイル判定)より先に**適用する。これによりリネーム(旧ファイル消失＋新ファイル出現)を**単一スキャン**で `旧行=missing` ＋ `新行=pending(candidate=旧行 id)` に確定できる(判定規則 3a は missing 化後の DB 状態を参照するため)。実装・凍結済み固定オラクル(工場非開示)・原典挙動(porting-spec RVP-SCAN-003 / 02§1 / 06§2)はいずれもこの順序で一致する(設計者注 — v2.0 表記是正: 旧文の実装クラス名・オラクル ID 直記を一般化)。番号は手順の論理的呼称であり実行順位ではない。

1. path 配下を列挙(include_subfolders=true なら再帰)。拡張子(小文字化)∈ {.jpg,.jpeg,.png,.gif,.bmp,.webp} のみ対象 (REQ-011)
2. 各ファイルの相対パス(スラッシュ統一、先頭末尾スラッシュなし、比較は case-insensitive)を算出 (REQ-014)
3. 各ファイルを以下の優先順で判定する(最初に成立した規則のみ適用)。**手順 4・5 を適用済みの DB 状態に対して**判定する(上記実行順序参照) (REQ-012):
   - (1) DB に同一相対パスの行があり、file_size と modified_date がともに一致 → 変更なし(スキップ)
   - (2) DB に同一相対パスの行があり、file_size または modified_date が異なる → SHA-256 再計算し
     hash/file_size/modified_date を更新(status は変更しない)
   - (3) DB に同一相対パスの行がない(新規) → SHA-256 計算:
     - (3a) 初回スキャン(last_scan が NULL)でない、かつ同一フォルダ内に同ハッシュ・status=missing の行が
       存在する → status=pending、candidate_link_id=その missing 行の id(複数一致時は id 昇順の先頭)で登録
     - (3b) それ以外 → status=normal で登録
     (created_date/modified_date はファイルのタイムスタンプの ISO 8601 UTC 変換)
   - 読み取り不能ファイル(ロック・アクセス拒否等)は DB を変更せずスキップし警告ログ。次回スキャンで再試行
4. DB 上 status=normal で物理ファイルが存在しない行 → status=missing へ更新(**手順 3 の判定より先に実行** — 上記実行順序参照)
5. 物理ファイルが存在しない status=pending の行 → 行を削除(候補が消えたため。**手順 3 より先に実行**)
6. 完了時(例外時も)last_scan を現在時刻に更新し、サマリ {added, missing, pending, updated, skipped} 件数を返す (REQ-015)

**ステータス遷移の全列挙 (REQ-016)** — 以下以外の遷移は存在しない:
| 遷移 | 契機 |
|---|---|
| (新規) → normal | スキャン規則 (3b) |
| (新規) → pending | スキャン規則 (3a) |
| normal → missing | スキャン規則 4(物理消失) |
| missing → normal | 再リンク確定(REQ-017。id 不変、パス情報を上書き) |
| pending → (行削除) | スキャン規則 5(物理消失)、または再リンク確定時の pending 側 |
| deleted | V1 では予約値。deleted へ遷移させる操作・deleted からの遷移は実装しない |

missing → pending の直接遷移は存在しない(pending は新規発見ファイル側にのみ付く)。
スキャンの二重起動は禁止(実行中は同一フォルダの再スキャン要求を拒否)。

**ハッシュ (REQ-013)**: SHA-256、ファイル全体、小文字 hex 64 文字。

**再リンク (REQ-017)**
- missing 画像を選択 → 同一 sync_folder 内・同ハッシュの pending 画像を候補として提示。
  複数候補は relative_path 昇順で列挙し、各候補に相対パス・ファイルサイズ・更新日時を表示する
- 確定前に確認(タグ・ノートが missing 側の画像として引き継がれる旨)を表示する
- 確定: missing 行に pending 行の relative_path/file_name/file_size/modified_date/created_date/hash を上書きし
  status=normal、candidate_link_id=NULL へ。pending 行は削除。**missing 側の image_id は不変**(タグ・関連保持が目的)
- 核/表面: core
- 受入観点: 遷移 4 ケース+再リンクを固定フィクスチャ(実ファイル+DB)で unit〜L2 exact 検査

### 2.2 タグシステム (REQ-020〜REQ-029)

- 3 種類の意味論 (REQ-020): simple は付いている/いないのみ(value=NULL)。textual は value に文字列。
  numeric は value に数値の不変文字列表現(InvariantCulture、比較は数値として行う)
- タグ名はシステム全体で一意・case-sensitive・空白のみは拒否 (REQ-021)。重複は DuplicateTagName エラー
- 階層 (REQ-022): 単一親。親削除→子の parent_id=NULL。循環(自己・子孫を親に指定)は拒否
- color は `^#[0-9A-Fa-f]{6}$` のみ受理、NULL 可 (REQ-023)。description は自由記述、NULL 可
- textual の predefined_values は順序保持の文字列配列。付与値はリスト外も許可(入力補助のみ) (REQ-024)
- numeric の min/max/step/unit は NULL 可。min/max 設定時は範囲外の付与を拒否(両端は含む) (REQ-025)
- 付与は UPSERT(再付与=値上書き、行は増えない) (REQ-026)。解除は行削除(冪等: 無い行の解除はエラーにしない)
- 一括付与・解除は単一トランザクション、失敗時全ロールバック (REQ-027)
- 削除カスケード (REQ-028): §2.0 の FK 規則どおり
- 一覧は name 昇順(大文字小文字を無視した序数比較=OrdinalIgnoreCase、同値時 id 昇順 — §4 掃討表と
  同一規則。v2.0 表記是正: 旧文「序数比較」は実装・掃討表と不一致だった)、使用数 = COUNT(DISTINCT image_id) (REQ-029)
- 核/表面: core(タグ編集 UI・色見本の描画のみ surface)
- 受入観点: 全規則 unit(exact、境界値ベクタ付き)。UI は golden+承認者

### 2.3 仮想ビュー (REQ-030〜REQ-033, REQ-038)

**CRUD (REQ-030)**: §2.0 のフィールド。name 必須(空白のみ拒否)。削除は条件・階層を連鎖削除。

**条件評価 (REQ-031)**: ビューの全条件を AND 結合し、status=normal の画像集合を絞り込む。
| operator | 対象 | 意味 | value / value2 |
|---|---|---|---|
| exists | 全 type | 当該タグが付与されている | 不使用 |
| equals | textual/numeric | value が完全一致(numeric は数値比較) | value=比較値 |
| between | numeric | value ≦ タグ値 ≦ value2(両端含む、数値比較) | value=下限, value2=上限 |
| regexp | textual | タグ値が正規表現にマッチ(.NET Regex、部分一致) | value=パターン |
| in | textual | タグ値が JSON 配列のいずれかと一致 | value=JSON 配列文字列 |

エッジケースの確定規則(いずれも例外でクラッシュしない):
- 対象タグが付与されていない画像 → いずれの演算子でも条件不成立(exists は false)。AND 結合のため当該画像は結果に含まれない
- simple タグ(value=NULL)への equals/between/regexp/in → 不成立。UI は simple タグに exists のみ提示する
- 数値変換できないタグ値(空文字含む) → equals/between の数値比較で不成立
- operator≠exists で value が NULL の条件 → その条件を評価から除外(無視)し警告ログ
- 不正な正規表現パターン・マッチタイムアウト(1 秒) → 条件不成立+警告(ログ+非モーダル通知)
- 実装方式(SQL/インメモリ)は自由。観測契約は §2.8 OC-1

**整列 (REQ-038)**: UI で選択可能な sort_field ∈ {name, modified_date, file_size}(既定 name。
created_date は v1.3/ECO-002 CR-4 で UI ソート軸から除外 — リスト基本列との整合・承認者裁定。
DB 既存値 'created_date' は後方互換として受理し従来どおり整列に使える)、
sort_direction ∈ {asc, desc}(既定 asc)。name は大文字小文字を無視した序数比較(OrdinalIgnoreCase 相当)。
二次キーは id 昇順(同値時も実行ごとに並びが変わらない安定ソート)。

**modified_at 規則 (REQ-032)**: ビュー本体・条件・階層のいずれの変更でも更新。閲覧では更新しない。

**一覧 (REQ-033)**: お気に入り=is_favorite=true を name 昇順。最近=modified_at 降順 limit 件(既定 10)。
いずれも同値時は id 昇順。
- 核/表面: core
- 受入観点: 演算子×型×境界値のテストベクタで unit(exact)。

### 2.4 タグ階層と NodeGraph (REQ-034〜REQ-037)

**階層定義 (REQ-034)**: ノード = {id, view_id, tag_id 必須, parent_id(NULL=ルート直下), position(同一親内 0 起点昇順),
alias(NULL なら tag.name を表示), condition_type ∈ {equals, range, pattern, values, NULL}, condition_value(JSON)}。
condition_value スキーマ: equals=`{"value":…}` / range=`{"valueFrom":…,"valueTo":…}` / pattern=`{"pattern":…}` /
values=`{"values":[…]}`。ノード移動で自己・子孫を親に指定→拒否。

**NodeGraph 構築 (REQ-035)**: 階層ノードを position 順に走査し、タグ type 別に展開する。
値の抽出対象は常に **status=normal の画像**のみ(REQ-016、INV-010):
- simple/numeric タグ → 1 ノード(配下に階層上の子ノードを接続)
- textual タグ → タグ名ノードの配下に「値ノード」を生成。値 = 当該タグが status=normal の画像に付与された
  distinct 値(predefined_values 外の自由入力値も含む。ノードに condition_type があればその条件で値を制限)、
  序数昇順
  - distinct 値が 0 件 → タグ名ノードのみ(値ノードなし)。ノードは表示・選択可能で、選択時は exists 条件と
    して評価される(結果 0 件なら空状態表示)
  - distinct 値が 1 件 → タグ名ノードと値ノードを統合した一体型ノード(表示名「タグ名: 値」)とし、
    階層上の子ノードは一体型ノードの配下に接続する
  - 2 件以上 → 値ノード群を生成し、階層上の子ノードは**各値ノード**の配下に接続する

NodeGraph は表示のたびにタグ付け状態から再構築する(キャッシュする場合はタグ付け・階層変更で無効化)。
タグ付けにより distinct 値数が 0↔1↔2 と変化した場合、次回構築で構造が変わる(選択状態は解決できなければルートへ)。

**絞り込み (REQ-036)**: 選択ノードまでのパス上の全ノードから条件列を生成し AND 評価(§2.3 の評価器を使用):
ルート=無条件 / simple・textual タグ名ノード=exists / 値ノード・一体型ノード=equals(その値) /
numeric ノード=condition_type に応じ equals または between(range の場合 valueFrom/valueTo)。

**ホームタグ (REQ-037)**: ビューを開いたとき home_tag_id(階層ノード id)が解決できれば該当ノードを初期選択。
解決不能(削除済み等)ならルート。エラーにしない。
- 核/表面: core(木構造の描画のみ surface)
- 受入観点: フィクスチャ(階層+タグ付け)→期待ノード木・期待画像集合の unit exact。値 0/1/2 件の境界必須

### 2.5 サムネイルと画像表示 (REQ-040, REQ-045)

- 生成 (REQ-040): 長辺 256px へ縮小(アスペクト比保持・拡大しない)。PNG 入力→PNG 出力、その他→JPEG 品質 80
  (品質値は実装規定 — L2 照合対象は寸法・形式のみ。v2.0 明確化)。
  保存先 `%APPDATA%/ViewPrism2/thumbnails/{MD5(画像絶対パスの小文字)}.{png|jpg}`(MD5 は小文字 hex —
  v2.0 明確化、実装と一致)。存在すれば再生成しない。
  生成失敗(壊れた画像)はキャッシュへ記録せずプレースホルダ表示とし(次回表示時に再試行)、スキャン・一覧表示を
  停止させない。読み取り不能なキャッシュファイルは削除して再生成する
- フルサイズ表示キャッシュ (REQ-045): メモリ LRU、上限 50 枚、有効期限 3 分(**絶対 TTL: ロード時刻
  起点。アクセスで延長しない** — v2.0 明確化、実装・既存 unit と一致)
- 核/表面: 生成パラメータ=surface(画像ライブラリ API へトレース)、キャッシュ規則=core
- 受入観点: 寸法・形式・キャッシュ動作は L2(出力ファイルのメタデータ検査)。LRU は unit(読込回数カウント)

### 2.6 シェルと画面構成 (REQ-041〜REQ-044, REQ-046, REQ-053, REQ-060) — v1.2 改訂・v2.0 GF 是正

**シェル**: 上部タブナビゲーション「タグ」「画像」+右端に「設定」。原典の「作業」タブは V3 スコープ
のため V1 では設けない。多重起動は禁止(2 つ目の起動は既存ウィンドウをアクティブ化して終了)。
単一ウィンドウ(§4 掃討表の同項と対応 — v2.0 で本文へ明記)。

**タグタブ(3 ペイン)** — タグ・ビュー・階層の管理:
- 左「ビュー管理」: 新規ビュー作成ボタン、ビュー一覧(各行: 名前+編集/削除アイコン)。選択中のビューが
  中央の階層編集対象になる。ビュー作成/編集ダイアログ = 名前(必須)+説明+お気に入り(v1.3 で実装・原典能力に整合)
- 中央「階層構造」: 選択ビューのタグ階層ツリー。タグパレットから D&D(またはボタン)でノード追加。
  ノード操作: 展開/折畳、ホームタグ設定/解除(設定中ノードは強調表示。**トグル・強調のアイコンは
  家アイコンを用いる。★は不可 — v2.0/REQ-060(c)、GF-03 是正**)、別名のインライン編集、
  条件設定(textual/numeric のみ。condition_type/condition_value のダイアログ)、削除。
  **ノードの条件サマリ表示は人間可読に整形する。書式は i18n リソースの条件型別テンプレートキーで
  定義し、ja 既定: equals=`値が一致: {value}` / range=`範囲: {valueFrom}〜{valueTo}`(片側のみの
  場合は `{valueFrom} 以上`・`{valueTo} 以下`)/ pattern=`パターン: {pattern}` / values=
  `いずれか: {値1}, {値2}, …`(区切りは `, `)。JSON 文字列や Unicode エスケープ(`男` 等)を画面に
  露出しない。格納値の JSON 形式(REQ-034)は不変 — v2.0/REQ-060(e)、GF-05 是正**。
  **編集はメモリ内で行い「保存」で一括コミット**(REQ-032 の modified_at 更新は保存時に 1 回)、
  「キャンセル」は確認後に破棄。未保存変更がある間のみ保存/キャンセルを活性化。
  **ノード追加ボタンは対象タグ名を表示する(例: 『「風景」をルートに追加』『「風景」を子に追加』。
  パレット未選択時は非活性+選択を促す文言)— v2.0/REQ-060(d)、GF-04 是正**
- 右「タグパレット」: 検索ボックス(名前の部分一致・大文字小文字無視)、「追加」ボタン→タグ作成ダイアログ、
  タグ一覧(色スウォッチ+名前+種類チップ+編集/削除)。**選択中タグの行は明確な選択視覚(背景強調+
  境界)で判別できること — v2.0/REQ-060(d)、GF-04 是正**。
  タグ作成/編集ダイアログ: タグ名(必須)・種類(シンプル/テキスト/数値、作成後の種類変更は不可)・
  カラー(プリセット+hex 表示)・説明。テキスト型: 候補値の追加・削除・D&D 並べ替え
  (**ダイアログは高さ固定とし、候補値の増加ではウィンドウを伸長させず候補値リスト内部のみ
  スクロールする — v2.0/REQ-060(a)、GF-01 是正**)。
  数値型: min/max/step/unit の入力

  Ctrl+クリック=トグル(選択順の番号バッジ表示)、**SHIFT+クリック=最後の選択アイテムからクリック位置までの
  一覧 index 範囲を既存選択へ追加**(union・置換しない。選択順は index 昇順で末尾へ追番。v1.3/CR-3 —
  原典 useImageSelection 方式)、**ダブルクリック=ビューア起動(タグ編集モード中は無効)**。

- リスト (REQ-042): display_columns(JSON 配列)に従う。列 = {type: basic|tag, key, label, width}。

- 詳細 (REQ-043): ファイル名/サイズ(1024 進、小数 1 桁、単位 B・KB・MB・GB)/解像度(px)/作成・更新日時

- タグ付与パネル (REQ-046): 「現在のタグ」(選択画像に付与済み。複数選択時は共通タグ。各行に解除×)と
  「タグを追加」(全タグ一覧+検索)の 2 区画。追加候補を選んで「適用」→選択画像全件へ一括付与(REQ-027 の
  原子バッチ)。textual は候補値ドロップダウン+自由入力、numeric は値入力ダイアログ
  (「全画像に同じ値」=固定値 / 「選択順に連番」=開始値+選択順 i)。範囲外値はダイアログ内で検証して拒否
- ビューア (REQ-044): 単一画像をウィンドウフィット表示(縮小のみ、拡大なし)。→/PageDown=次、←/PageUp=前、
  Esc/閉じるボタン/**画像外余白のクリック**=閉じる(背景クリックは原典準拠 — モーダルオーバーレイから
  ウィンドウ方式への適応。v1.3/CR-7。**v2.0: 余白クリックで閉じるは normal モードのみ有効** — REQ-054、
  scroll/spread ではページ送りクリックとの衝突防止のため無効)。並びは呼び出し元一覧の整列結果。
  端で停止(ループ・例外なし)。表示中はタイトル等に「現在位置/総数」を表示。
  **v2.0: 本項の単一画像表示はビューア表示モードの normal に相当する。拡張モード(scroll/spread)と
  共通設定・モード切替は §2.9(REQ-054〜059)** — Esc・閉じるボタン・現在位置/総数・端停止は全モード共通

空状態の規則(全画面共通):
- グリッド/リスト: 画像 0 件 → 中央に「画像がありません」相当のメッセージ(i18n キー)。何もない場所の
  ダブルクリックは無視

### 2.7 i18n と設定 (REQ-050〜REQ-052)

- 2 言語(ja 既定/en)。リソースはキー→文言の辞書。欠落キーは ja へフォールバック、それも無ければキー文字列を
  返す(例外を投げない) (REQ-050)。文言資産は原典 messages/ja.json・en.json から該当分を変換(K-BOM 資産)
- 言語切替は即時反映(再起動不要)+ settings.json へ永続化 (REQ-051)
- settings.json(%APPDATA%/ViewPrism2/): locale、ウィンドウ位置・サイズ・最大化状態、表示モード(グリッド/リスト)、
  最後に開いたビュー id、最後に選択したコレクション id(v1.3/CR-5・6。グリッド列数キーは CR-1 で廃止 —
  残存しても無視)。破損時は既定値で起動し、ファイルを再生成 (REQ-052)
- 核/表面: フォールバック・永続化=core。文言・書式=surface(翻訳資産へトレース)
- 受入観点: フォールバックと破損耐性は unit exact。表示言語は golden+承認者

### 2.9 ビューア拡張モード (REQ-054〜REQ-059) — v2.0 新設

V1 の単一画像ビューア(§2.6 REQ-044)を 4 つの表示モードへ拡張する。REQ-044 の規則(起動経路・
Esc/閉じるボタン・現在位置/総数表示・端停止・呼び出し元一覧順)は全モードの共通基盤として不変。

**キーボード規則の共通基盤**: REQ-044 の「→/PageDown=次、←/PageUp=前(1 画像)」は normal と
scroll にそのまま適用される。見開き(spread)のみ矢印キーを REQ-057 の規則(ステップ・右開き反転)で
再定義する。**PageDown/PageUp は全モードで論理キー(常に 次へ/前へ・見開きでは現行ステップ分)であり、
読書方向で反転しない**。

**表示モードとモード切替 (REQ-054)**
- モード: normal(既定。§2.6 REQ-044 の単一画像表示そのもの)/ scroll / spread-right / spread-left
- ビューア上部のコントロールバーにモード切替と表示設定(REQ-058 の resizeMode/alignMode/gapMode+
  customGapPx の数値入力・ページ送りモード・空白ページ開始)の UI を集約する(細部レイアウトは
  K-DESIGN の範囲で製造判断、golden G-8 で承認)
- モードごとに最後に表示した画像 index を記憶し、モード切替時にそのモードの前回位置を復元する。
  **各モードの記憶の初期値はビューア起動時の index(クリックした画像)** — 未訪問モードへの初回切替は
  起動時の画像から始まる(原典準拠)。記憶はビューアを開いている間のみ(閉じると破棄。永続化しない — REQ-059)
- 画像外余白クリックで閉じる(REQ-044)は normal のみ。scroll/spread では無効(ページ送りクリックと
  衝突するため。Esc・閉じるボタンは全モード有効)

**スクロールモード (REQ-055)**
- 画像を一覧順(REQ-038 整列結果)で縦一列に表示。水平は中央揃え。画像間隔は gapMode(REQ-058)
- **画像サイズ**: 各画像は表示領域の幅に収める(幅超過時は縮小・拡大はしない・アスペクト比維持 —
  REQ-044 のフィット規則の幅適用)。横スクロールを生じさせない。縦方向は画像の自然な高さ(縮小時は
  比例縮小後の高さ)で積む
- **クリック**: スクロールモードでは画像・余白のいずれのクリックも操作を持たない(閉じない・送らない。
  誤操作防止 — REQ-054)
- 段階読み込み: 全件を一括読み込みしない。現在位置の前後ウィンドウを読み込み、上端・下端への接近で
  範囲を拡張する(ウィンドウ幅・接近閾値は実装定数 — E-BOM で確定し固定オラクルの合否対象にしない)
- 現在位置の追跡: ユーザーのスクロール停止後、表示領域の垂直中央に最も近い画像(画像矩形中心と
  表示領域中心の距離最小。**同距離の場合は一覧 index の小さい方**)を現在位置とする(OC-11)。
  この追跡が「現在位置/総数」表示とモード別位置記憶(REQ-054)に反映される
- ←→キーまたは外部からの位置変更(モード復元含む)では、対象画像の位置までスクロールする
- 個別画像の読み込み失敗は当該画像の位置にエラー表示(ファイル名+失敗の旨)を出し、前後の閲覧を
  継続できる(ビューア全体を壊さない — porting-spec 05§5 検収項目)

**見開きモード (REQ-056, REQ-057)**
- ページ配置(OC-9): spread-right(右開き・日本式)は 右=現在 index・左=現在+1。
  spread-left(左開き)は 左=現在 index・右=現在+1。ペア相方が末尾を超える場合は空白ページ
  (無地。エラー表示・文言を出さない)
- 空白ページ開始(startWithEmptyPage、既定 OFF): ON のとき index 0 では最初の画像を進行方向側
  (右開き=左ページ / 左開き=右ページ)に配置し反対側を空白に(表紙を単独で見せ、以後 index (1,2)(3,4)…の
  ペアを揃える — 0 起点 index)
- ページ送り(OC-10): ステップ = SHIFT 押下時 1 / 通常時 pageTurnMode(doublePage=2 既定、singlePage=1)。
  - キーボード: spread-right は →=前へ・←=次へ(進行方向反転 — 紙の右開き本と一致)。
    spread-left は →=次へ・←=前へ。**PageDown/PageUp は論理キーとして方向反転せず常に 次へ/前へ**
    (REQ-044 共通規則の継承。v2.0 確定 — 沈黙次元 §4)
  - ページクリック: **クリック領域は表示領域全体を左右半分に二分した各半面**(画像の上・画像外の
    余白・空白ページを含む面全体)。進行方向側の半面(右開き=左半面 / 左開き=右半面)=次へ、
    反対側の半面=前へ(原典準拠 — 見開きに「閉じる余白」は存在しない。REQ-054)
  - 境界: 前へは現在-step<0 なら 0 へ。次へは現在+step>総数-1 なら総数-1 へ、既に総数-1 なら何もしない。
    空白ページ開始 ON・index 0・2 ページ送りの「次へ」は index 1 へ(以後の偶奇を揃える)。
    末尾クランプの帰結として、最終画像がペアの相方として見えている状態からさらに「次へ」を送ると
    最終画像が単独ペア(最終, 空白)として再表示される — **意図的(原典準拠)**: 最終画像を読み位置として
    確定する遷移であり、エラーやループにしない
- 画像読み込み失敗はその面にエラー表示(もう片面と操作は継続可能)

**共通表示設定 (REQ-058)** — 適用対象をモードで限定する(原典実装準拠)。表中の**太字が既定値**:
| 設定 | 値(既定) | 適用対象 | 意味 |
|---|---|---|---|
| resizeMode | matchLargerHeight / matchSmallerHeight / **noResize** | 見開きのみ | ペアの高い方/低い方の画像高さへ両者を統一(OC-12)。統一高さは表示領域高さの 90% を上限。**片側が空白ページの場合(match 系のみの規則)は有効画像 1 枚の自然高さを統一高さとする(90% 上限は適用)**。noResize は原寸で「統一高さ」概念を持たない(片側空白でも同じ。表示領域超過時は領域内に収める=上限 100%。match 系 90% との非対称は原典準拠の意図的差)。アスペクト比は常に維持 |
| alignMode | top / **middle** / bottom | 見開きのみ | ページ内の垂直揃え(スクロールは対象外 — 原典仕様書記載はあるが実装が存在せず、実装準拠で不採用と確定) |
| gapMode (+customGapPx) | **tight** / loose | 見開きの左右間+スクロールの画像間 | tight=間隔 0(見開きでは左右画像を中央境界へ寄せる)。loose=customGapPx の間隔(customGapPx 既定 0 — loose+0px は tight と視覚同一になってよい) |

normal モードは V1 のウィンドウフィット表示(REQ-044)を維持し、上記設定の対象外。

**設定の永続化 (REQ-059)**
- mode/resizeMode/alignMode/gapMode/customGapPx/pageTurnMode/startWithEmptyPage を settings.json
  (§2.7 REQ-052 の機構)に保存し、次回ビューア起動時に復元。変更は即時保存
- customGapPx は 0〜100 の整数。範囲外・型不正の保存値は既定 0 として読む(破損耐性は REQ-052 準拠)
- モード別位置記憶(REQ-054)は永続化しない

- 核/表面: ペアリング・送り・追跡・高さ統一・位置記憶の各計算=core(OC-9〜13 として ViewModel 単体で
  観測可能にする)。描画・スクロール体感・設定 UI=surface
- 受入観点: 計算ロジックは unit exact(OC-9〜13 のベクタ)。設定永続化は unit ラウンドトリップ
  (REQ-052 の既存検査へ追加)。画面・操作感は golden G-8+承認者。大量画像のスクロール性能は
  探索プローブ(合否に入れない)

### 2.10 類似検索とマージ (REQ-061〜REQ-067) — v3.0 新設

重複画像を pHash で見つけて 1 枚に統合する。検出系(pHash 計算・距離・検索)は核、操作系(検索 UI・
マージ・トラッシュ)は表面。スコープは検出+マージまで(00-charter-v3.md §裁定記録): **detailed/ORB・
criteria 条件検索・relink 修復・トラッシュの復元/完全削除は本ループ対象外**(§6 カバレッジ監査)。

**2.10.1 知覚ハッシュ pHash (REQ-061, OC-14)**

pHash は決定的な DCT ベースのアルゴリズムで算出する。原典(sharp ベース)とのビット完全一致は要求しない
(CPOL-103: pHash は preserve_with_adapter)。本書のレシピを当方の正準とし、Phase 3 ADR で確定・pin する。
レシピ(spec 固定 — 決定性の根拠。下記の数値規約はすべて Phase 3 ADR で pin する):
1. デコード後の画像を 32×32 にリサイズ(アスペクト比は無視し正方化、補間は**双線形**に固定)し、
   グレースケール化する(輝度 y = 0.299·R + 0.587·G + 0.114·B を計算し、**`(int)(y + 0.5)` で 0.5 切り上げ**して
   8bit 整数化する。.NET 既定の銀行丸め(ToEven)は使わない — 決定性のため明示固定)
2. 32×32 の輝度行列に **直交正規化形の 2 次元 DCT-II**(orthonormal DCT-II)を適用する。行方向→列方向の
   2 パスで計算し、cos テーブルは**事前計算した固定定数表**を用いる。基底・符号規約は ADR で一意に固定する
   (中央値比較 (c>m) はスケール不変だが符号規約が違うとビットが反転するため固定が必須)
3. 出力行列の左上 8×8(行 i・列 j、0≤i,j<8 → 係数 [i][j])= 低周波 64 係数を取り出す
4. DC 係数 [0,0] を除いた 63 係数の中央値 m を求める(DC は全体輝度を表し支配的なため中央値計算から除外)
5. 8×8 の各係数 c(DC 位置も含む)について bit = (c > m) ? 1 : 0 を**行優先(row-major、[0,0]→[0,7]→[1,0]…)**で
   64bit に並べる。**DC 位置のビットは c=DCT[0,0] >> m でほぼ常に 1 となる低エントロピービット**であり、
   64bit 長を保つための意図的設計(実効情報量は概ね 63bit)
6. 64bit を最上位ビットから 16 桁の**小文字 16 進**で表現する

- リサイズ補間方式・グレースケール係数と丸め・DCT 規約・cos テーブルは決定性の生命線であり spec/ADR で
  固定する。SkiaSharp は版差で出力が変わり得るため**版を exact ピン**(53-service-bom)し、リサイズの
  サンプリング指定を明示する(K-BOM/ADR で確認)
- pHash は画像のピクセル内容のみに依存し、ファイル名・パス・更新日時には依存しない
- 縮退ケース: 単色画像など全係数が等しい場合、c=m となり手順 5 は bit=0 に倒れる(全 0 に近い pHash)。
  受入ベクタの代表として明記する
- 受入(OC-14): (a)同一ピクセル内容→常に同一 pHash(決定性)、(b)単色・既知パターン画像の手計算可能な
  pHash ベクタ(縮退ケース含む)、(c)同一内容=距離 0・微小変化(再エンコード/わずかな縮小)=閾値内の小距離・
  明確に異なる画像=大距離。16hex の網羅ベクタは手書きせず(b)(c)の性質ベースで凍結する
  (固定オラクル S-19 系 — 41-fixed-oracle で実装・本書外で閉じる)

**2.10.2 距離と類似度 (REQ-062, OC-15)**

- 距離 = 2 つの pHash の**ハミング距離**(64bit XOR の立ちビット数、0〜64)。popcount で算出
- 類似度% = 距離の単調非増加な区分線形関数(spec 固定のブレークポイント)。
  ブレークポイント (距離, 類似度%): **(0,100) (5,90) (10,70) (15,50) (20,30) (25,10) (64,0)**
- **変換式(一意・決定的)**: 距離 d が含まれる区間 [d_lo, d_hi](対応 [s_lo, s_hi])に対し
  `raw = s_lo + (s_hi − s_lo) · (d − d_lo) / (d_hi − d_lo)`(double 演算)を計算し、
  **`sim = (int)Math.Floor(raw + 0.5)`(0.5 切り上げ=AwayFromZero)で整数化、[0,100] にクランプ**する。
  ブレークポイント上の距離はその区間の端点 s をそのまま採る。距離が増えれば類似度%は決して増えない(単調非増加)
- **期待値ベクタ(OC-15 / 固定オラクルの正)**: d=0→100, 3→94, 5→90, 7→82, 10→70, 12→62, 15→50,
  20→30, 25→10, 26→10, 27→9, 40→6, 64→0(最終区間は緩勾配のため丸め後に同値が連続し得るが非増加は保つ)
- この変換は pHash のビット表現に依存しない(距離値のみの関数)
- 受入(OC-15): 全ブレークポイント+上記期待値ベクタ+各区間の丸め境界をテストベクタで exact(unit)

**2.10.3 特徴量・類似度の永続化とキャッシュ無効化 (REQ-063, OC-18)**

- **image_features**(列: image_id PK、phash、file_size、modified_date、hash、last_calculated)。
  pHash 等の特徴量を画像ごとに保存。ORB 列(orb_descriptors/orb_keypoints)は**本ループでは作らない**(ORB defer)
- **image_similarity**(列: cache_key PK、image_id1、image_id2、similarity_score、last_compared)。
  画像ペアの類似度キャッシュ。**ペアは文字列比較で小さい方を image_id1・大きい方を image_id2 に正規化**し、
  **cache_key = `{image_id1}-{image_id2}`**(本ループは pHash 単一モードのためモード識別子は付けない。
  ORB 導入時に `-{mode}` 接尾辞を追加する将来拡張余地のみ残す)。(A,B) と (B,A) は同一キャッシュ
- **無効化は内容ベースのみ**: 対象画像の file_size / modified_date / hash のいずれかが image_features 記録と
  異なれば特徴量を無効として再計算する。**時間ベース失効(原典の 24h TTL)は採用しない**(決定性・単純性。
  内容変化の検知で十分 — §4 第3回掃討)
- **連鎖無効化**: 画像の特徴量(pHash)を再計算した場合、その画像が image_id1 または image_id2 に含まれる
  image_similarity 行を**削除**する(古い類似度を返さない)。次回検索時に再計算・再保存する
- 画像削除時は FK CASCADE で両テーブルの関連行を削除する
- 受入(OC-18): 特徴量の保存→取得ラウンドトリップ、内容変化での再計算+関与ペアの連鎖無効化、ペア正規化と
  cache_key 生成、CASCADE 削除を exact(unit/L2)。マイグレーション適用後スキーマ(列・FK・索引・既定値)を
  新規作成 DB と一致(REQ-004 の L2 検査へ追加)

**2.10.4 類似検索 (REQ-064 エンジン=核 / REQ-065 UI=表面, OC-16)**

- エンジン: 基準画像 1 枚に対し、**同一コレクション内**(REQ-053)の **status=normal の画像のみ**を候補とし
  (基準画像自身は除外)、各候補との pHash 類似度を算出して**閾値以上(≧)**(既定 70・範囲 50〜100・整数刻み)を
  **類似度降順(同値時は id 昇順で安定)**で返す。
  **候補の status フィルタ(normal のみ)はキャッシュ参照より先に適用**し、非候補(deleted/missing/pending)は
  キャッシュ値があっても結果に含めない。フィルタ後の候補について特徴量・ペア類似度をキャッシュ(2.10.3)で
  参照し、無ければ計算して保存する。検索は UI をブロックしない(非同期実行・進捗通知が可能)
  - **候補を normal に限定する根拠(v3.0 — G2 補正)**: V3 のユースケースは「重複 normal 画像を見つけて統合」。
    missing は物理ファイルが無く pHash を新規計算できない。pending/missing を候補にする検索は relink 修復の
    用途であり後続ループ(V4)へ defer(§6)。これにより INV-010(既定の表示・抽出は normal のみ)とも整合する
- UI: 一覧/グリッドで画像 1 枚を選択して起動。閾値スライダー(50〜100・整数・既定 70)を持つ。
  **本実装の照合は pHash のみ**であり、原典が持つ精度モード(simple=pHash / detailed=pHash+ORB)の**選択肢は
  表示しない**(RVP-REPAIR-005/§裁定記録 4 — 効かない設定の誤解防止。ORB は後続ループ)。
  結果は類似度%付きで降順に一覧表示し選択できる。検索中は進捗、結果 0 件は空状態を表示
- 性能(1,000 枚規模)は探索プローブ P-07 対象(原典は 1,000 枚で約 1.1 秒・5,000 枚でフリーズ感の記録 —
  42-exploratory-probes で実装・合否に入れない・本書外で閉じる)
- 受入: エンジンは OC-16(候補抽出=normal 限定・自身/コレクション境界・閾値境界(≧)・安定ソートを
  固定フィクスチャで exact、unit)。UI は golden G-9+承認者

**2.10.5 マージ (REQ-066 規則=核 / REQ-067 UI=表面, OC-17)**

- 規則: マージ先(保持)1 枚とマージ元(統合)1 枚以上を指定する。マージ先・マージ元はいずれも
  **status=normal を前提**とし、マージ先とマージ元が同一の指定・マージ元の重複は拒否する。
  操作(単一トランザクション・原子 — INV-006):
  1. **タグ集約(union)**: マージ元のタグをマージ先へ付与する。両方に同一タグ(同一 tag_id)がある場合は
     **マージ先の値を保持**し、**マージ先の値が NULL/空でマージ元に値があればマージ元の値を採用**する
     (RVP-MERGE-002=マージ先優先・§裁定記録 3)。simple タグ(値なし)は存在の union のみ。
     付与は UPSERT(REQ-026)に従い (image_id, tag_id) は高々 1 行(INV-003)
  - **マージ元が複数の場合の決着順**: マージ元を **id 昇順**で処理する。あるタグについてマージ先が NULL/空の
     とき、**最初に非空値を与えたマージ元の値を採用**し、以降のマージ元の同タグ値では上書きしない
     (決定的・id 昇順先勝ち)
  2. **マージ元の status を deleted にする**(論理操作)。マージ元の image_tags は削除しない
     (deleted ステータスで一覧から除外されるのみ。完全削除/復元は後続ループ)。
     deleted 化したマージ元の image_features/image_similarity 行は残置してよい(候補は normal 限定=2.10.4 の
     ため検索に混入しない)
  3. **物理画像ファイルは一切移動・削除・変更しない**(INV-009 woven 安全制約)。本ループはこの制約の
     **初の実アクション適用**である
  - 失敗時は全ロールバック(部分適用を残さない)
- UI: 類似検索結果または複数選択からマージ先(1 枚)とマージ元(1 枚以上)を指定して実行できる。実行前に
  マージ先・マージ元と統合後タグのプレビューを示し、**物理ファイルは削除されない旨を明示**する。
  マージで deleted になった画像が「見えないまま消える」ことを避けるため、最小のトラッシュ(deleted 画像一覧)
  表示を提供する(**選択中コレクション内**(REQ-053)の deleted 画像の一覧と件数のみ。**復元・完全削除の操作は
  後続ループ** — RVP-STATUS-003/004)
- 受入(二経路は相補・両方必須):
  - **核**: OC-17(タグ union 衝突=マージ先優先・NULL 補完・多元 id 昇順先勝ち・simple union・status 遷移・
    原子性を unit exact)
  - **物理非破壊**: **L3 物理差分 CP-MERGE-001**(操作前後でマージ元・マージ先のファイルのバイト列と mtime が
    不変=自動の固定検査。33-control-plan で定義・**本書外で閉じる**)
  - **表面**: マージ UI・統合タグプレビュー・トラッシュ一覧は golden G-9+承認者(承認者の目視確認。L3 CP の
    自動実証とは別経路で、視覚・操作の妥当性を担保する)

- 核/表面: pHash 計算・距離変換・検索エンジン・マージ規則・キャッシュ無効化=core(OC-14〜18 として ViewModel/
  サービス単体で観測可能にする)。検索 UI・マージ UI・トラッシュ表示・進捗描画=surface
- 受入観点まとめ: 核は unit exact(OC-14〜18)+マージの物理非破壊は L3 物理差分。表面は golden G-9+承認者。
  大規模類似検索の応答性は探索プローブ(合否に入れない)
