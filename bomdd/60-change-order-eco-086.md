# Change Order — ECO-086(staged): ビュー階層ノードの定義値展開モード+タグ定義の閉じた値集合属性

- 起票日: 2026-07-14 / 報告者: maintainer(設計方針はメンバー助言を起票前相談で整理)
- 種別: 新機能(NodeGraph 展開意味論の拡張=観測値展開と別系統の「定義値展開」新設)
- baseline: main `350d345`

## 1. 要求(maintainer・2026-07-14)

現行のビュー軸ブラウズで textual タグをノードに置くと、**付与済みの distinct 値**(観測値)が
値ノードとして展開される(REQ-035)。これは有用だが、都道府県(候補値 47)のようなタグでは:

- 47 個の条件ノード手書きは非現実的(現行の代替手段が実質ない)
- 観測値展開は付与状況依存で、0/1/2+ 件によりノード構造自体が変わる(REQ-035 の一体型統合)
- 展開順が付与状況・序数順任せで、定義順(北海道→沖縄)にならない

これに対し、**タグに定義された値ドメインを、付与状況にかかわらず・定義順で・安定して展開する**
「定義値展開」を新設したい。要求の骨子(起票前相談でメンバー助言を取り込み整理):

1. **ビュー階層ノードに展開モードを導入**: Manual / ObservedValues / DefinedValues /
   DefinedAndObservedValues。DefinedValues では textual の predefined_values を定義順で、
   画像 0 件でも全値ノードを生成する(構造安定)。
2. **numeric タグの定義域展開**: min/max/step(REQ-025。例: 評価 1-5 step 1)から値ノードを
   生成する。現行 numeric は単一ノードのみで、評価=5 のリーフを作るには condition_type 付き
   ノードを 5 個手書きするしかない(textual 47 条件と同じ非現実問題が numeric 側にもある)。
3. **タグ定義側に「値集合の意味」属性を新設**: 候補値が入力補助(現行 REQ-024=自由入力可)か、
   閉じた値集合かを区別する。閉集合のタグでは候補外値(例: 都道府県=「蝦夷」)を通常の
   48 番目ノードとして並べず、**未定義値として検出**する。
4. **多段の定義値展開**: 都道府県→評価 の階層で、展開された各都道府県値ノード配下に評価の
   定義値リーフ(1-5)が生成され、パス選択(北海道→評価 5)が
   「都道府県 Equals 北海道 AND 評価 Equals 5」に変換されることを回帰テストで固定する。

### 責務分担の方針(起票前相談で整理・gate① で確定)

| 置き場所 | 持つもの |
|---|---|
| タグ定義(textual/numeric settings) | 値集合の意味(入力補助/閉集合)・定義値・順序・候補外値の扱い |
| ビュー階層ノード(view_tag_hierarchies) | どの値源から展開するか(展開モード)・0 件ノードの表示 |

## 3. 切り分け済みの事実(2026-07-14 コード読解・grep で確定)

**確定(実測済み)**:

- **numeric タグは値展開されない**: `NodeGraphBuilder.ExpandNode`
  (src/ViewPrism2.Core/Services/NodeGraphBuilder.cs:190-196)は `tag.Type != TagType.Textual`
  で単一 TagName ノードを生成するのみ。「都道府県→評価」を現行で書いても各都道府県の下は
  「評価」1 ノードで、評価 1-5 のリーフは生成されない。
- **定義順維持は値源差し替えでは実現しない**: `ExtractValues`(同 :256-271)は値源が何であれ
  最後に `.Order(StringComparer.Ordinal)`(序数昇順)を通す。`TagValueIndex.FromValues` に
  定義順リストを渡してもソートで潰される(序数順だと京都が北海道より先に来る)。
  → 展開ロジック本体の改変が必須=「値源フックの差し替えだけ」では済まない。
- **`ITagValueSource` 契約は観測値専用**: インターフェース仕様(同 :10-14)に
  「status=Normal の画像に付与された distinct 値のみ返すこと(INV-010)」と明記。定義値を
  この契約に流し込むのは契約違反ハック。定義値供給は**別契約**(または明示的なモード付き
  契約への改版)として設計する。
- **0/1/2+ 件の構造分岐は仕様**(REQ-035 の一体型統合・同 :200-229)。DefinedValues の
  「0 件でも構造安定」はこの分岐のモード別迂回=明示的な仕様追加が必要(REQ-035 の改版では
  なく併存する新 REQ とし、既存観測値展開の意味論は不変に保つ)。
- **パス→条件変換は改変不要の見込み**: `PathConditionConverter`
  (src/ViewPrism2.Core/Services/PathConditionConverter.cs:32-34)は `NodeKind.Value` を
  タグ型に関係なく Equals(その値)へ変換する。numeric の定義値リーフが Value ノードとして
  生成されれば「北海道→評価 5」は既存ロジックのまま AND 2 条件になる。
- 候補値の器は既存: `textual_tag_settings.predefined_values`(順序保持 JSON 配列・REQ-024)、
  numeric は `numeric_tag_settings` min/max/step/unit(REQ-025)。定義値の**データ源**は
  新設不要で、不足はその**解釈**(入力補助か閉集合か)と**展開経路**のみ。

**疑い(未検証・fix 時にプローブで確定)**:

- 評価器(OC-1)の numeric Equals が Value ノード由来の文字列値で正しく効くか
  (REQ-031 の数値変換比較との整合。回帰テスト固定対象=§1 要求 4)。
- ECO-084 の表示モード(REQ-094 すべて/未分類)と 0 件ノードの相互作用
  (0 件ノードは未分類件数に寄与しない=自明のはずだが、チップ件数表示の実装が
  空 matched 集合を正しく扱うか)。
- パッケージ(REQ-093)/スナップショット(REQ-092)が textual_tag_settings・
  view_tag_hierarchies の新列を搬送するか(形式バージョン繰り上げの要否)。

## 4. 是正方針(案 — gate① 裁定+着手時確定)

1. **CAD(ViewPrismUI)**: tag_tab.md(タグ定義モーダル=値集合の意味スイッチ+候補外値の
   扱い/階層エディタ=ノードの展開モード設定)・image_tab.md(ビュー軸での定義値展開の
   見え方・0 件ノード・未定義値の検出表示)+mock 改版。
2. **仕様**: §2.2(タグ=値集合の意味属性)・§2.4(定義値展開の新 REQ 群: モード 4 値・
   定義順・0 件安定・numeric 定義域展開・閉集合の未定義値検出・既定=ObservedValues)。
3. **DB**: `view_tag_hierarchies` に展開モード列、`textual_tag_settings` に値集合の意味
   (+numeric 側の要否は裁定)。マイグレーションは ECO-020 方式が前例。既存行の既定=
   ObservedValues 相当(既存ビュー完全互換)。
4. **実装**: NodeGraphBuilder のモード分岐(定義値供給の別契約+定義順保持+0/1/2 分岐の
   迂回)・階層エディタ/タグ定義モーダル UI・ビュー軸表示。PathConditionConverter は
   無改変見込み(プローブで確認)。
5. **オラクル**: 既存固定行不変(R6)。新規行=定義順 47 展開・画像 0 件での構造安定・
   numeric 1-5 リーフ生成・「北海道→評価 5」の AND 変換・閉集合の未定義値検出・
   既存ビューの既定モード互換・DefinedAndObservedValues のマージ順序。

### gate① 裁定論点 — ✅ 裁定済み(2026-07-14 maintainer: a〜f すべて案のとおり承認)

| # | 論点 | 裁定(=案のとおり) |
|---|---|---|
| a | 展開モード体系の採否と語彙 | **承認**: Manual / ObservedValues / DefinedValues / DefinedAndObservedValues の 4 値。既定=ObservedValues(既存ビュー互換)。condition_type は「観測値の制限」として現行維持し、Manual との直交性整理は CAD/仕様化で確定 |
| b | 閉集合×既存候補外付与値の扱い | **承認**: 検出警告(表示のみ)。付与時拒否はしない=閉集合化した時点で既に付与済みの候補外値はデータとして保持し、検出表示で気づかせる |
| c | 未定義値の検出表示 | **承認**: ビュー軸で未定義値を通常ノードと区別して検出表示(専用「未定義」ノード/警告バッジ=形態は CAD で確定)。未分類(タグなし・ECO-084)との概念区別を UI で表現 |
| d | 0 件ノード表示 | **承認**: 既定=表示(構造安定が本機能の目的)+ビューノード側「0 件を隠す」オプションを設ける |
| e | numeric 定義域の境界 | **承認**: min/max/step いずれか NULL の numeric に DefinedValues 指定=展開不能として警告+ObservedValues フォールバック。step 数過大時の上限を設ける(具体値は仕様化で確定) |
| f | DefinedAndObservedValues のマージ順序 | **承認**: 定義値を定義順で先・観測のみの値(開集合タグの自由入力値)を末尾に序数順 |
