## 1. 症状

ViewPrismUI の UI 部品表 `docs/04_component_registry.md` の **CMP-008 ContextPaneChrome
(右ペイン/モーダル共通クローム)** の契約が

> **契約**: ヘッダ固定／本体のみスクロール／docked アクション常時可視(内容が何件増えても押し出されない)。

の 3 点のみで、ECO-116/117 で確立し golden 承認済みの **「その本体スクロールは末尾に到達できる」**
が含まれていない。出典として挙げている画面 CAD(`docs/screens/image_tab.md` 行107)は
ECO-119 の是正でこの句を明記しているため、**部品層への転写時にだけ脱落**している。

部品表は「契約は本書が持ち、画面 CAD は部品名+差分だけを書く」(冒頭「目的と位置づけ」)という
**実装側が参照する正典**として新設された(ViewPrismUI `78c95a7` で恒久運用規則を刻印)。
したがってこの脱落は、次に CMP-008 を参照して製造される面が ECO-116/117 と同じ欠陥
(スクロール容器自身の `Padding` で内容末尾が隠れる)を再生産する経路になる。

## 3. 切り分け済みの事実

### 確定(証拠あり)

1. **契約に無い**: `04_component_registry.md:209` の契約 3 点に末尾到達性の記述がない。
   同ファイル全体を `スクロール|overflow` で grep した結果、末尾到達性に触れる行はゼロ。
2. **出典には有る**: `docs/screens/image_tab.md:107`(ECO-119 是正後)は
   「同パネルに適用されるのは『ヘッダ固定／本体スクロール』であり、その**本体スクロールは
   末尾に到達できる**ことを含む(ECO-116)」と明記。CMP-008 行215 はこの行を証拠として引きながら
   当該句を落としている = **転写時の脱落**であって、契約の意図的な縮小ではない。
3. **適用面で実証済みの欠陥クラス**: CMP-008 の適用面に挙がる**ゴミ箱ポップアップ**(work_tab L57)は、
   ECO-117 で `ScrollViewer.Padding=18` により**実測 18px 到達不能**だった当該面
   (`ImageTabView.axaml:1463` / `WorkTabView.axaml:1123`)。
   部品の適用面自身に実測実績があるのに、部品契約がそれを言っていない。
4. **契約喪失の経路が開いている**: 部品表の運用方針は「既存の画面 CAD は **lazy 遡及**:
   次に各面へ触れる fix 時に部品参照へ置換する」。この方針どおり `image_tab.md:107` が
   将来 CMP-008 参照へ置換されると、**末尾到達性の記述はどの CAD にも存在しなくなる**。
5. **VP2 側は健全**: `33-control-plan.yaml` の CP-UI-G1 に ECO-116/117 の到達性観点が刻印され、
   CP-AXAML-LINT-117 が `ScrollViewer.Padding` のゼロ基線を機械的に維持している。
   つまり **ViewPrism2 の実装層は守られているが、CAD 部品層だけが弱い**(片側だけの防御)。

## 4. 是正方針(案・着手時確定)

### 案A(推奨) — CMP-008 の契約へ末尾到達性を刻む+実装写像を紐づける

1. CMP-008 の**契約**へ 4 点目を追加: 「**本体スクロールは末尾に到達できる**
   (スクロール容器自身の内側余白で内容末尾を隠さない)」。
2. **実装写像の注記**を添える: Avalonia では `ScrollViewer.Padding` は Viewport から引かれず
   Extent とも一致しないため内容末尾が `Padding.Top` のぶん到達不能になる
   (実測 16/14/18px の三点確認 = ECO-116/117)。余白は内容側要素の `Margin` で持つ。
   機械的維持は ViewPrism2 の **CP-AXAML-LINT-117**(ゼロ基線 lint)。
   — これは **ECO-107 教訓1 / ECO-117 教訓1「注意書きは掃射手段を紐づけない限り規約として機能しない」**
   の予防適用(部品表の記述にも掃射手段を紐づける)。
3. **証拠欄**へ ECO-116/117 を追記(現状は ECO-119 のみ)。
4. **掃射**(ECO-117 教訓): 部品表内で内部スクロールを契約に含む他部品は CMP-006 のみ
   (起票時 grep 済み)。CMP-006 は Provisional・契約未収束のため、
   同クラスの適用先である旨を **Candidate 記帳(REG-C4)** に留める(契約改版はしない)。
5. 画面 CAD 側(未検証 A)は**本 ECO では触れない**。委譲関係の成立は lazy 遡及方針どおり
   各面に触れる fix の仕事(触ると本 ECO のスコープが画面 CAD 全面へ広がる = R3)。

diff 規模: CAD 1 ファイル・数行。ViewPrism2 側 src/tests/BOM は無変更。

### 案B(却下) — 画面 CAD の各面へ個別に末尾到達性を追記

面ごとの転写 = 部品表が明文で禁じる「黙って面ごとの値として転写しない」(権威規則3)に反し、
面間非対称の潜伏を作る。ECO-116/117 が実証した欠陥の発生機序そのものなので却下。

  CP-AXAML-LINT-117 が既カバー)。**本 ECO の再発防止は部品表側そのもの**= 契約 4 点目+
  実装写像注記(掃射手段の紐づけ)+REG-C4 記帳(ECO-120 前例の型)。
- **教訓(read-across 明記)**:
  1. **共通契約層への転写は「脱落」として弱化しうる — 転写の忠実性は過剰と過少の両方向で破れる**。
     ECO-119 は横展開時に存在しない実例を**作話**(過剰)し、本件は共通層への転写時に承認済み条項を
     **脱落**(過少)させた。双対で N=2。転写を検収する際は「書かれたことが正しいか」(作話検査)と
     「書かれるべきことが揃っているか」(被覆検査=出典の条項単位突合)の両方が要る。
  2. **共通層の弱化は lazy 遡及と組み合わさると契約の消失に進行する**。部品表の運用(画面 CAD は
     次に触れる fix で部品参照へ置換)の下では、共通層が出典より弱いまま置換が進むと、脱落条項は
     どの文書にも存在しなくなる(§3-4)。共通層を検査の照合先へ昇格させる変更(ECO-122)の前に
     共通層自身の被覆を検査する順序が正しかった。
  3. **「照合先の昇格」は正本の検収イベント**。検査器が読むものを変える時(ECO-120= validate_bom の
     検査範囲・ECO-122= probe の照合先)は、新しく読まれるものの健全性検査が前提整備になる
     (ECO-120「検査器の検査範囲は読んでいるファイル一覧でしか保証されない」の CAD 版)。

## 9. human gate の提示(起票時)

CMP-008 は **Standard** 部品であり、部品表の運用規則は「**Standard の確定・改版は maintainer 裁定**」
と定めている。一方、本件で追加する条項は **ECO-116/117 で既に golden 承認済みの契約の転写**であり、
部品表初期投入時の方針(「初期投入はすべて既存 CAD/mock/golden 裁定からの**転写**=新規発明ゼロ」)
と同じ性質である。したがって:

- **転写と読む場合** → 裁定不要。`/eco-fix eco-121` で着手できる。
- **Standard 改版と読む場合** → gate①(maintainer 裁定)が必要。

**AI の推奨 = 転写(裁定不要)**。理由: ①追加内容は新規発明ではなく既存 golden 承認済み契約
(ECO-116/117)の再登録 ②適用面(ゴミ箱ポップアップ)で実測済み ③契約を**強める**方向のみで
既存の適用面・実装に一切の変更を要求しない ④現状を放置すると lazy 遡及の進行に伴い
契約が CAD から消失する(§3-4)。
