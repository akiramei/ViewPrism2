# ECO-130 — スキャンの二段階化 — 即時 DB 反映を廃し、ステージング→遷移別サマリー→ユーザー適用/破棄へ(implemented)

- 報告者: maintainer 設計まとめ(2026-07-21「画像の状態管理・あるべき状態管理」§8〜§12)の投入。3 分割の (c)
- 種別: 仕様改訂候補(スキャン適用モデルの変更+UI 新設。実装逸脱ではない)
- baseline: ViewPrism2 main `390e1f4`
- 関連: **ECO-129 の前提**(推奨実施順序の先頭)/ ECO-128 と同一設計まとめ起点

## 1. 症状(変更要求)

現行スキャンは走査しながら 512 件バッチで逐次 DB commit する一段階方式。USB HDD 未接続・
別 HDD が同一ドライブ文字・走査対象の誤りで、確認機会なく大量の状態変更(missing 化)が確定する。
設計まとめ §9 はこれを二段階へ:

```text
1. ファイルシステムを走査
2. DB との差分を計算
3. スキャン結果サマリーを表示(遷移別件数)
4. ユーザーが適用または破棄
5. 適用時だけ状態を更新
```

- サマリーは遷移別(normal→missing / normal→pending / 新規 pending / deleted 除外 / 読み取り失敗 …)(§10)
- missing 率は**注意表示のみ**(グリーン/イエロー/レッド+件数・割合・説明文)。システム判断には使わない(§11)
- 変更**件数**に比例した確認強度(適用ボタンへの件数表示・大規模時は確認ダイアログ)(§12)
- 用語: スキャン途中=「キャンセル」/ 完了後=「結果を破棄」(§9)

## 3. 切り分け済みの事実

### 確定(証拠あり)

1. **一段階の実体**: 512 件バッチ逐次 commit([ScanService.cs:23](../src/ViewPrism2.Infrastructure/Scanning/ScanService.cs))・
   missing 化は判定に先行して即 DB 反映(:148-176)・**キャンセルしても commit 済みバッチは残る**
   (ct.ThrowIfCancellationRequested はバッチ間)= §9 の「スキャン完了前に 1 件ずつ更新しない」と正反対。
2. **増分公開との衝突(性能設計)**: ECO-059/060 が 26 万件対応のために導入した
   `ScanBatchCommitted` 増分公開(:341-348「唯一の公開境界: repository transaction が成功してから通知」)は
   「commit = 公開」を前提とする。二段階化は公開境界を「適用時」へ移す再設計=
   **26 万件経路の性能規律**(経路棚卸し様式 N=7・ECO-125 Recompute 台帳)での設計が必要。
   ステージング差分の保持形(メモリ/一時テーブル)も 26 万件で成立する形を選ぶ。
3. **読み取り失敗の混在**: 読み取り不能ファイルは skipped++ で「変更なし」と同一カウンタ
   (:225-230)。§10 の「読み取り失敗」独立表示には集計の分離が必要。
4. **§8 の部分対応は既在**: ルートフォルダ不存在時は一括 missing 化せず中断(:111-115)。
   ただし「別 HDD が同一ドライブ文字」「中身だけ空」は現行では防げない= 二段階化の残存動機。
5. **ScanSummary の現行形**: {added, missing, pending, updated, skipped} の事後報告のみ。
   遷移別(normal→missing / pending→missing 等)・deleted 除外件数・修復候補数はない。
6. **OC-5(判定器)は器として温存可能**: ScanJudge は純粋関数(判定と適用が既に分離)。
   二段階化は「判定結果を即適用せず蓄積する」だけで判定器の構造とは整合的
   (ECO-129 による判定内容の変更とは独立の軸)。

### 未検証(疑い・着手時確定)

- ステージング中のアプリ操作との整合(スキャン中にタグ付け・整理・ゴミ箱操作が走った場合の
  差分の鮮度。適用時の再検証要否)。
- 二重起動ガード(_inFlight)とステージング破棄の相互作用。
- 進捗表示(ECO-060 の「総件数不明の間は percent を捏造しない」)のステージング版への引き継ぎ。

## 4. 是正方針(案・着手時確定)

段階: **CAD mock(ViewPrismUI)→ spec 改訂(手順・REQ)→ 実装追随**。

- ScanService: 走査+判定を「差分計算」(DB 無変更)と「適用」(単一の適用イベント)へ分離。
  ScanJudge は不変(ECO-129 と合流する場合は判定内容のみ変わる)。
- サマリー面: 遷移別件数・missing 率注意表示(3 色+件数/割合/説明文)・件数入り適用ボタン・
  規模比例の確認強度(確認ダイアログは CMP-011/ECO-126 様式)・「キャンセル/結果を破棄」の用語分離(i18n)。
- **missing 率でシステムが適用禁止・自動破棄をしない**(§11 の原則を spec に明文化)。

diff 規模: 大(スキャン経路の構造変更+UI 新設)。ECO-129 と別軸だが spec §2.1 を共に触るため、
実施順序は本 ECO 先行(または同時改訂)を推奨= 適用前確認なしの新規全 pending 化(129)による
裁定爆発を防ぐ(maintainer 指示済み)。

## 8. 停止点= golden 合格基準(gate②・実機。26 万件プロファイル推奨)

1. **初回スキャンの無傷(SCAN-001 回帰)**: 新規フォルダを登録→スキャン。従来どおり確認なしで
   取り込まれ、段階的公開(一覧へ逐次追加・完了時ソート)が従来どおり動く。
2. **再スキャンの二段階(正常系)**: 既存コレクションを再スキャン → SC-1(差分計算・処理済み件数が
   進む・DB 無変更の注記)→ 完了で遷移別サマリー。ファイルを数枚 削除/変更/追加した状態で件数が
   内訳どおりか(適用ボタンのラベル件数= 変更合計)。
3. **破棄= 無変更**: サマリーで「結果を破棄」(または ✕)→ 一覧・件数・最終スキャン日時が一切
   変わらない。再スキャンすると同じ差分がもう一度出る。
4. **適用= 反映**: 適用 → コレクション管理の行メッセージにサマリー・一覧へ反映(missing 化した画像が
   既定一覧から消える・新規が現れる)・最終スキャン日時更新。適用後の修復画面に missing が並ぶ。
5. **キャンセル= 無変更**: 大きめのコレクションで SC-1 中に「キャンセル」→ 行メッセージ
   「スキャン結果を破棄しました(変更なし)」・DB 無変更。
6. **missing 率カードと安全側(REQ-100 の核)**: USB 未接続相当(フォルダを一時リネーム等)で再スキャン
   → レッドのカード(件数・割合・説明文)が出て、**適用ボタンは有効のまま**・「結果を破棄」で無傷。
   ※ ルートフォルダ自体が無い場合は従来どおり即エラー(本画面は一瞬で閉じ行メッセージへ)。
7. **確認強度**: 変更 1,000 件以上を作って適用 → SC-6 確認ダイアログ(キャンセル/N件の変更を適用する)。
   確認中にサマリー側の破棄・✕ が効かない(再入防止)。適用中も ✕ で閉じない。
8. **詳細面**: 変更 100 件以上で「変更内容を確認」→ 遷移別グループ+「…ほか N 件」→ 戻る。
9. **視覚**: mock(VPUI captures/scan_summary/SC-1〜6)との並置確認。**判断材料(§7a R7 の残差分)**:
   ①SC-1/SC-6 の幅(実装= 640 統一/ConfirmDialog 420。mock 作画= 480) ②SC-4 でも作業量 callout が
   出る(mock は未作画) ③遷移語彙が現行判定準拠(「メタ更新(状態不変)」「新規 normal」等=
   ECO-129 で mock の pending 語彙へ収束予定) ④ja/en 両言語(ECO-079 規約)。
10. **回帰**: 修復(missing→relink)・ゴミ箱・整理マージ・エクスポートが従来どおり。

合格なら `/eco-accept eco-130` を指示してください(基準 9 の許容裁定も添えて)。
不合格所見(GF-*)は本 ECO の手順 1 から。
