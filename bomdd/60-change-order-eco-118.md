# ECO-118: タグ付与/剥奪の応答劣化 — 選択規模の変更が母集合規模の再読・再構築を通る(26 万件で顕在化)

- 起票日: 2026-07-20
- 報告者: maintainer(実機観測)
- baseline: main `1dc1f5c`
- 種別: 不具合是正候補(性能・実装層。視覚/意味論不変)

## §1 症状

画像タブのタグ編集モードで、選択画像へタグを付与・剥奪すると**待たされることがある。
26 万画像のビューで約 2 秒**(2026-07-20・maintainer 実機)。

報告者の診断規範: タグ編集は**選択画像のみ**に影響するため、コストがビューの画像数
(母集合)に比例するのはおかしい — ECO-113(選択クリック)と同じ「期待計算量との乖離」の
言語化(ECO-113 教訓 2 の再適用)。

**26 万件経路の 6 経路目**(064 起動/062 検索/113 選択/114 モード遷移/115 パネル操作に続く)。
ECO-114 R8 の cheat-log 予告記帳(2026-07-19)は「ApplySequential 後の再読込等」を同族として
挙げており、ECO-115 は「タグ付与/剥奪系= データ変化につき Recompute 正当」として**明示的に
対象外へ送った**。本 ECO はその残余の顕在化= R3 分離起票の実行。

## §2 工程診断(R2)

| 工程 | 判定 | 根拠 |
| --- | --- | --- |
| CAD(ViewPrismUI) | **健全(非該当)** | 付与/剥奪の意味論(行クリック即時付与・チップ件数の追随・REQ-094 未分類離脱)は CAD 契約どおりで、変更しない。性能は NFR 層 |
| BOM | **契約の適用範囲外** | E-UI-BROWSE-022 の計算量 invariant(ECO-113/114/115)は「母集合不変の操作」を対象にしており、**データ変更系の計算量契約が無い** → fix 時に「変更コストは変更集合サイズ比例」を刻印 |
| 実装 | **逸脱確定(両面・§3.1)** | 選択規模の変更に対し O(母集合) サイト 3 つ+O(全 DB 付与行) 1 つのコード証拠 |

結論: **実装層の性能欠陥**。裁定(gate①)は不要です。

**ECO-115 の切り分けの精密化**: ECO-115 が「Recompute 正当」としたのは**意味論**
(データ変化後に表示が追随すべき)であり、**O(母集合) の実装コスト**ではない。
意味論上再解決が必要なのは「影響を受けた画像の帰属・表示」と「チップ件数」だけで、
どちらも選択規模の差分で解決できる。

## §3 切り分け済みの事実

### 3.1 確定(起票時・コード実測)

付与/剥奪の全経路(シンプル行クリック/候補値チップ/数値セル/連番/現在タグ ✕)は
`ReloadTagsAsync` に合流する([ImageTabViewModel.cs:2171-2176](../src/ViewPrism2.App/ViewModels/ImageTabViewModel.cs))。
その内訳が全部母集合規模:

| # | サイト | コスト | 選択規模との乖離 |
| --- | --- | --- | --- |
| 1 | `RefreshImageTagsAsync`(:555)= `GetImageTagsByFolderAsync(コレクション)` | **コレクション全付与行の DB 再取得+全再グループ化**(26 万件級) | 変わったのは選択画像の行だけ |
| 2 | `BuildEntries`(:567) | **全 26 万 ImageEntry 再構築**+辞書再構築 | タグリストが変わったのは選択分だけ |
| 3 | `Recompute`(:2175) | ECO-114 §3.1 の全パイプライン= 全件条件評価×(1+子チップ数)+全件ソート+**26 万 ImageItemVM 再構築+ObservableCollection 26 万 Add** | 表示が変わるのは選択分+チップ件数だけ |

- **WorkTab 同型(read-across 面・DB スコープはより広い)**:
  `ReloadTagsAsync`([WorkTabViewModel.cs:1037-1043](../src/ViewPrism2.App/ViewModels/WorkTabViewModel.cs))=
  **`GetAllImageTagsAsync()`= DB 全体の付与行を再取得**+Recompute。母集合(workspace)は
  小さいが、26 万件コレクションが存在する DB では**タグ 1 枚の付与で全 DB 付与行を読む**。
- **混入**: `6f7b4f9`(M3a 初版・2026-06-18)= ECO-113/114/115 と同根・同日。潜伏約 1 ヶ月。
- **マスキング**: 通常規模では体感不能。26 万件系 5 ECO はすべて「母集合不変の操作」を
  スコープにしており、データ変更系は ECO-115 が意味論を根拠に対象外へ送った(§2)。

### 3.2 意味論の分解(何が本当に再解決を要するか)

変更されたデータ= **選択画像の image_tags 行のみ**。表示上の帰結:

1. **選択画像の表示**(タグドット/現在のタグパネル/タグ追加行の Added)— O(選択)。
2. **チップ件数**(ビュー軸の子ノード件数・未分類件数)— 差分加算 or チップ構築のみ再実行。
3. **ビュー軸の帰属**(REQ-094: 未分類モードで付与→離脱・タグビュー条件の出入り)—
   **影響を受けた画像だけ**現ビュー条件を再評価すれば足りる(挿入/除去はソート済み列への
   二分挿入= O(log n + 移動))。FS 軸では帰属不変。
4. **ソート位置** — ソートキーが当該タグ値に依存する場合のみ、**選択分の**再配置。

すべて選択規模の差分で閉じる= O(母集合) は純粋な無駄。

### 3.3 未検証(fix 時に実測)

- 3 サイトの寄与割合(2 秒の内訳。ECO-113 §同型の fix 時計測)。
- チップ件数の「差分加算 vs チップ構築だけ再実行」のどちらで足りるか(構築コスト実測次第)。
- ソートキーのタグ依存判定の要否(v2 ソートは表示列基準= タグ列ソート中の付与で位置が動く
  経路が実在するか)。

## §4 是正方針(案・着手時確定)

**案 A(推奨・差分更新)**: `ReloadTagsAsync` を選択規模の差分経路に置換:

1. **差分再読**: 選択画像の image_tags だけ再取得(per-image API か WHERE IN。無ければ
   Core への読み口追加は互換構造化= ECO-086 前例)。
2. **差分エントリ更新**: `BuildEntry` を選択分のみ再実行し `_entries`/`_entryById` を
  in-place 置換(ECO-113 の辞書資産を再利用)。
3. **表示の差分更新**: 該当 ImageItemVM のみ更新(ECO-114 の観測可能化・ドット常時構築資産を
   再利用)。ビュー軸の帰属変化は選択分だけ条件再評価して Items へ挿入/除去。
4. **チップ件数**: 差分加算 or チップ構築のみ再実行(§3.3 の実測で裁定)。
5. **WorkTab 同時是正**: 全 DB 取得→workspace/選択スコープの再読へ(§8.2 閉包)。

probe= 構造 probe(付与/剥奪が母集合列挙・全再構築を通らない= Items インスタンス同一性+
ECO-113 計器)+意味論維持(REQ-094 未分類離脱・チップ件数追随・ドット/Added 反映)。
