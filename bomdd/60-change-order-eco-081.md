# ECO-081 — テスト実行の時間上限をハーネス自身の宣言へ(dotnet test 間欠ハングの fail-closed 化+正本乖離解消)

- status: staged
- type: 工程ハーネス是正(検査器の fail-closed 化。製品 src 不変=ECO-061/078/080 同型)
- baseline: main 988a01c
- 発端: maintainer 工程考察(2026-07-13)=「テスト実行タスクが時間内に完了することを信用できないのは問題。現対策(結果のファイル書き出し+監視+タイムアウト)は観測の工夫であって根本解決ではない」

## §1 症状

`dotnet test tests/ViewPrism2.Tests` が**間欠的に返らない**(全テスト緑でもプロセスが終了しない)実績があり、運用は「ハングしたら exe 直接実行」という但し書きつきになっている。AI サブタスクでの実行でも応答なしのまま放置されるケースを maintainer が複数回目撃。検査に逃げ道の但し書きが必要な時点で、**検査器の fail-closed が破れている**。

構造的問題: テスト実行の時間上限が**誰の契約にもなっていない**。呼び出し側(AI・maintainer・将来の CI)がそれぞれ私的なタイムアウト+ログ監視を持つ構図で、ハーネス自身は無限に待ち得る。

## §3 切り分け済みの事実(確定と疑いを分離)

### 確定(実測)

- ハーネス構成: xunit.v3 **3.2.2** + Microsoft.Testing.Platform(`TestingPlatformDotnetTestSupport=true`・ViewPrism2.Tests.csproj)。**xunit.runner.json / testconfig.json は不在**=タイムアウト・診断系の設定はゼロ(2026-07-13 実測)。
- xUnit v3 ネイティブ exe 直接実行(`ViewPrism2.Tests.exe`)は 2026-07-13 の複数回実測で毎回 ~13 秒で確実に返る。
- 「ハング」の一部はシェル側バッファリング(`Select-Object -Last` の全バッファ)と診断済み(maintainer 2026-07-07)=ログファイル経由で解決済み。**本 ECO の対象は残りの testhost 側**。
- 本セッション(2026-07-13)の `dotnet test` は全 run 完走=間欠事象であり、いま赤いプローブは作れない。

### 疑い(未検証)

- testhost 非終了の機序: `dotnet test` は testhost の graceful exit を待つため、テスト完了後の残存非バックグラウンドスレッド(容疑: Avalonia Headless の Dispatcher/セッション・SQLite 接続プール・background Task・タイマー)があると返らない。exe が安定なのはランナーが明示終了するため — **ダンプ未実測の仮説**。間欠のため、先に上限+診断採取を恒久化し、次の発火時に証拠が自動的に残る状態を作るのが実測主義に合う(プローブ先行の変形)。

## §4 是正方針

**原則: タイムアウトを呼び出し側の防御からハーネス自身の宣言へ。**

1. **A. 時間上限+超過時診断の恒久宣言**: テスト実行に上限時間を設定し、超過時は**残存スレッドの診断つきで失敗**させる(「無限に待つ」→「時間超過で証拠を吐いて赤」)。実装手段は fix 時に xunit.v3 3.2.2+MTP で**実際に効くものを実測選定**(候補: MTP のタイムアウト系オプション/設定ファイル・xunit 診断設定・hang ダンプ採取)。上限値はフル run 実測 ~13-25 秒に対し十分な余裕(例: 5 分级)を置き、フレークでの誤爆を避ける。
2. **陽性対照(ECO-078/080 教訓=予防ゲートには赤プローブが存在しない)**: 人工的にハングするテスト(無限待ち)を一時的に注入し、上限が実際に発火して診断つき失敗になることを**実測**してから恒久設定として出荷する(注入テスト自体は残さない or skip で封印=手段は fix 時裁定)。
3. **C. 正本乖離の解消**: CLAUDE.md 機械受入の `dotnet test` 規定を実態に合わせて改訂(タイムアウト付き `dotnet test` を正とするか、exe 直接を正式経路として併記するかは fix 時の実測結果で裁定)。memory の実行規律とも整合させる。
