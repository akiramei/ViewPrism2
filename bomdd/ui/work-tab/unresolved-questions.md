# 作業タブ 未解決質問(UQ-W*)

CAD(`ViewPrism2 作業タブ.html`)抽出時点の設計フォーク。**UQ-W01 と UQ-W05 は E-BOM/M-BOM/schema の形を決めるため、UI-BOM/ECO 着手前に maintainer 裁定が要る。** 他は私の lean を併記。

## UQ-W01 — 作業スペースの永続化モデル【maintainer 裁定・最重要】

作業スペース = 名前付き・永続・ユーザー管理の画像集合。**第一級ドメインデータ**であり config ではない。

- **(A) DB テーブル化(lean)**: `workspaces(id, name, is_default, seq, created_at)` + `workspace_images(workspace_id, image_id, added_at)` 結合表。既存 `sync_folders`/コレクションと同型。再起動跨ぎ・件数/所属クエリ・参照整合・スケールに自然。schema migration + `WorkspaceRepository` + `WorkspaceService` が要る。
- **(B) settings.json(AppSettings)**: workspaces を JSON 配列で保持。migration 不要で軽いが、画像 id リストを設定に持つのは SettingsStore の「小さな構成」役割に不適合・参照整合なし・デフォルトスペースが多数の画像参照を抱えうる。

**lean = (A)**。理由: 名前付き画像集合は collection/sync-folder と同格のドメインで、既存モデリングに倣うのが自然。確定すると M-BOM(32)・control-plan(33)・schema・固定オラクル(S-3x 候補)が決まる。

## UQ-W02 — 作業対象の受け渡し結線【maintainer 裁定】

画像タブ作業モード「追加」(ECO-017 `AddToWork`)は現状 `_workTargets`(session `List<string>`)へ蓄積するだけで**行き止まり**(ECO-017 が永続化/受け渡しを「別ループ」へ繰延)。本タブの成立で「追加」=**デフォルト作業スペースへの書き込み**へ変える。

- 論点: 「追加」直後にトースト/ステータス通知を出すか(どのスペースに入ったか)。lean = ステータスバーに「N 枚を作業スペース『…』へ追加」。
- 論点: workTargets チップ(画像タブ ECO-017 の「作業対象 N 枚」)は受け渡し後どう振る舞うか。lean = デフォルトスペースの累計を反映(または撤去し作業タブへ集約)。

## UQ-W03 — ツールバー「整理」を本タブで出す【決定 2026-06-29 maintainer】

モックは「整理」ボタンを描画するが DCLogic に整理状態が無く onclick 未配線=**プレースホルダ**。

**決定(2026-06-29 maintainer)**: **作業タブに整理ボタンは必要=出す。** 作業タブ=雑多な作業のためのスクラッチ空間。「似た画像を作業スペースに集めて、そこで吟味して整理(マージ)する」のは代表シナリオ。整理は「重複の見極めとマージ」という*作業そのもの*で、少数候補に絞った作業スペースはマージ前レビューに最適。「整理は取り込み時=画像タブの責務」という当初前提(私の旧 lean=出さない)は否定。

- **(B) 出す = 確定**: E-UI-SIMILARITY-035/E-UI-MERGE-036(ECO-014 整理トレイ)を作業タブへ文脈モードとして配線。プレースホルダの「整理」ボタンを本配線。→ 作業タブは右ペイン surface を一段広く再利用。
- (A) 出さない: **不採用**。

将来導線(モック外・深追いせず): 画像タブ整理「似た画像を探す」の結果を**作業スペースへ送る**入口 → 「探す(画像タブ)→集める(作業スペース)→マージ(作業タブ)」が一本化。
[[mock-ui-ir-is-cad]] 描画モック基準では「描画あり・機能なし」だったが、maintainer が描画意図=将来配線の予約と確認。

## UQ-W04 — ゴミ箱確認文言の INV-009 是正【是正で確定・要承認】

モック確認文言「ゴミ箱内の N 枚の画像を完全に削除します。**元のファイルも削除され**、この操作は元に戻せません。」は **INV-009(物理非破壊)違反**。image-tab **ECO-019 T1 が同一文言を既に是正済**(「画像ファイルは削除されません(DB から除去)・元に戻せません」)。

→ 作業タブのゴミ箱は同一 surface の再利用ゆえ**同じ是正を適用**(woven 安全制約は CAD より上位)。新規裁定ではなく既決則の適用。

## UQ-W05 — スコープ/ECO 分割【maintainer 裁定】

本機能は過去の画像タブ ECO 群(Core 意味論を再利用するだけ)より大きく、**net-new Core(WorkspaceService/Repository)+ schema + 新タブ**を含む。

- **(A) 単一 ECO**: Workspace Core + 永続化 + 作業タブ shell + サイドバー + 受け渡し結線 + 右ペイン surface 再利用 を一括。
- **(B) 段階分割(lean)**:
  - **ECO-α**: Workspace ドメイン Core + 永続化 + 作業タブ shell + サイドバー(作業スペース CRUD・デフォルト回転・選択・件数)+ 受け渡し結線(MVP=スペースに画像が集まり閲覧できる)。
  - **ECO-β**: 右ペイン surface(タグ編集/作業=別スペース移動/⋯=修復・削除・ゴミ箱 popup)を作業タブへ再利用配線。
  - **ECO-γ**(任意): 整理(UQ-W03 で (B) を採る場合のみ)。

lean = (B)。理由: net-new Core を先に固定オラクルで固め、surface 再利用は別段で golden に回せる(画像タブの M1→M4 段取りと同じ規律)。
