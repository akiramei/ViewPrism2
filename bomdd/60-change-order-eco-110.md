# ECO-110: 右ペイン開閉時のグリッド作業対象喪失 — スクロールアンカー契約の未定義

- 起票日: 2026-07-18
- 報告者: maintainer(UX 所見)
- 種別: 不具合是正候補(UX・CAD layoutInvariant の保存対象曖昧 → gate① 裁定要)
- status: staged

## §1 症状

画像タブ(グリッド表示)で、右ペインが出る文脈モード(タグ編集モード・整理モード)を
開始すると、一覧の幅が縮む(右ペイン=Dock Right・幅 344px)ことでグリッドが再流動し、
**直前まで見ていた画像がビューポートから消える**。

体験シーケンス(報告文の再構成):

1. ユーザーはグリッドを見ながら「この画像たちにタグ付け/整理をしよう」と意思決定する。
2. モード変更ボタンを押下 → 右ペインが表示され、中央グリッド幅が 344px 縮む。
3. UniformGridLayout の列数が減り全行が再流動。スクロールオフセット(ピクセル値)は
   保持されるが、**同じオフセットが指す内容が変わる**ため、見ていた画像がスクロールアウト。
4. ユーザーは作業対象を見失い、スクロールで探し直す。

- 対象: グリッド(アイコン)表示のみ。リスト表示は行が全幅・行高固定のため、幅変化で
  行の再流動が起きず、オフセット保持=内容保持が成立する(症状なし)。
- 終了時も同型(右ペインが消えて幅が広がる → 逆方向の再流動)。

## §3 切り分け済みの事実

### 3.1 確定(起票時実測)

- 右ペインは Dock Right・固定幅 344(ImageTabView.axaml:110 `EditMode` / :314 `IsOrganizeContext`)。
  開閉はオーバーレイでなく**中央幅の増減**として作用する。
- グリッドは `ScrollViewer > ItemsRepeater + UniformGridLayout(MinItemWidth=164)`
  (ImageTabView.axaml:1049-1052)。幅 −344px ≒ 2 列減 → 全行再流動。
- コードビハインド・VM とも、モード遷移時にスクロール位置へ触れる処理は皆無
  (保持も補正もしない= Avalonia ScrollViewer の素の挙動でオフセットが残るだけ)。
- 閲覧(browse)時は選択不可(ECO-017: 既定選択はタグ編集・作業の選択許可モードで有効化)。
  したがって**モード開始時点では「作業対象」は選択状態として存在せず、視認位置だけが手掛かり**。
- リスト表示は症状なし(§1)。作業タブは workspace 型(ペイン幅可変)だが本症状の報告面は
  画像タブのみ — 同族面の悉皆は裁定後の fix でスコープ確定(R3: 本 ECO で無断拡大しない)。

### 3.2 疑い(未検証)

- **CAD mock(HTML)はブラウザ標準のスクロールアンカリング(CSS `overflow-anchor:auto` 既定)
  により、右ペイン開閉時も先頭可視要素が自動維持される可能性が高い**。真なら CAD 行87 は
  暗黙に「可視コンテンツ保持」を意図しており、mock では無記述でも自然充足していたことになる
  (=実装だけが弱読み)。mock 実挙動は未実測 — 裁定時に maintainer が mock で確認可能。
- 開閉アニメーション有無・レイアウト確定タイミング(Avalonia の Measure/Arrange 後に補正が
  必要)は fix 時のプローブで確定。

## §4 是正方針(案・gate① 裁定対象)

いずれも CAD 行87 の精緻化(ViewPrismUI 側改稿)→ BOM 宣言 → 実装の順。

### 案A: 先頭可視アイテムアンカー(推奨)

右ペイン開閉(開始/終了とも)の直前にビューポート先頭の完全可視アイテムを記録し、
再レイアウト後にそのアイテムが先頭付近に来るようオフセットを補正する。
ブラウザ標準スクロールアンカリングと同等= mock の自然挙動(§3.2 疑い)とも整合。

- CAD 改稿: 行87 に「保存対象=可視コンテンツ(先頭可視アイテムのアンカー)。ピクセル
  オフセットの保持は充足と見なさない」を追記。
- 実装: ImageTabView.axaml.cs にモード遷移フック+アンカー実測/補正(推定 50〜80 行)。
- diff 規模: 中(コードビハインド 1 面+headless probe 2〜3 本)。VM/Core 無変更。
- golden: 実機シナリオ 1〜2 点(開始時/終了時に視認画像が可視のまま)。R7 captures は
  動作契約のため対象外(静止画で検証不能)— probe が機械受入の主体。

### 案B: 案A+作業対象優先

選択アイテムがある場合は先頭可視より選択先頭の可視維持を優先(BringIntoView)、
なければ案A へフォールバック。

- ただし §3.1 の通り**モード開始時は選択が存在し得ない**(閲覧=選択不可)ため、
  案A との差分は実質**終了時**(選択を残したまま閲覧へ戻る遷移があるか次第)のみ。
- diff 規模: 案A+小。効果の増分が薄い可能性 — 開始時の症状(報告の主訴)には案A で十分。

### 案C: 現状追認(オフセット保持を契約と明文化)

CAD 行87 を弱読み(a)に固定し、報告された体験問題を容認する。**推奨しない**
(報告の主訴を仕様化で棄却することになる。mock 自然挙動(§3.2)が真なら CAD とも矛盾)。

### サブ裁定: 適用範囲

- **(a) 右ペイン開閉のみ(推奨)**: 症状起点に最小スコープ。
- (b) 中央幅が変わる全事象へ一般化(左ペイン折畳み 276↔64=行83 の同族・ウィンドウ
  リサイズ)。効果は大きいが検証面が広がる — (a) の効果確認後に拡張起票が安全。
