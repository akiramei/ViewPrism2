# /eco-file — ECO 起票+工程診断

典拠: [bomdd/change-management.md](../../../bomdd/change-management.md) §3。
引数: 症状・要求・所見の自由文(再現手順や観測があれば含める)。

## 絶対規律

- **このスキルの中で src/tests のコードを変更してはならない**(R1)。変更は /eco-fix の仕事。
- 診断は証拠(ファイル・行・コミット)で確定させる。推測は「疑い(未検証)」として事実と分離して書く(ECO-037 書式)。

## 手順

1. **採番**: `bomdd/60-change-register.yaml` 末尾で次番号を確認(逐次採番)。
2. **症状/要求の記録**: 再現手順・観測日・報告者。事実と疑いを分離。
3. **工程診断(R2)** — 修正対象を決める前に、欠陥(または変更)がどの工程に属すか判定する:
   - **CAD(ViewPrismUI)**: `docs/screens/*.md` と mock に定義があるか。未定義・曖昧なら CAD 欠陥。
   - **BOM(bomdd/30-ebom, 32-mbom)**: surface/unit の宣言と受入観点は健全か。
   - **実装**: CAD・BOM が健全で実装だけが逸脱しているか。該当コードとバインド先を読み、
     可能なら混入コミットを `git log -S` で特定する(潜伏期間とマスキング要因も記録)。
   - 未確定事項(FL-*/VE-* 等)との関係を必ず確認: 未実装/仕様未確定をバグと誤認しない。
4. **起票文書作成**: `bomdd/60-change-order-eco-NNN.md`。書式は ECO-037/038 に倣う:
   §1 症状 / §2 工程診断(表) / §3 切り分け済みの事実(確定と未検証を分離) /
   §4 是正方針(案・着手時確定) / §5 影響 BOM / §6 残ゲート。
5. **register エントリ追加**: `status: staged`・baseline(現 HEAD)・findings・impacted_bom・body。
6. **検証+コミット**: `python bomdd/validate_bom.py` 0 error を確認し、
   `起票(eco-NNN): <要約>` でコミット(pre-commit が validate を再実行する)。

## 停止点(human gate の提示)

診断結果に応じて、**人間がやることだけ**を明示して停止する:

- **実装逸脱と確定** → 「裁定は不要です。/eco-fix eco-NNN で是正に着手できます」
- **CAD 未定義・曖昧** → 裁定が必要な選択肢を diff 規模・golden 影響つきで提示し、
  「ViewPrismUI 側の裁定(gate①)をお願いします」で停止。CAD 是正が先、製品コードは後。
- **検査器/台帳の欠陥** → doc-only 是正の内容を提示。

## スコープ外所見(R3)

診断中に別の問題を見つけたら、**現 ECO に混ぜず**、その場でもう 1 件 /eco-file 相当の
分離起票を行うか、`bomdd/51-cheat-log.md` に所見として記録する。ついで修正は禁止。
