# ADR-0003: データアクセスは Microsoft.Data.Sqlite + Dapper(手書き SQL)を採用する

- 状態: 承認済み(2026-06-11)
- 決定者: 設計AI(委任範囲)

## 文脈
仕様(REQ-003〜005)は SQLite の物理的な契約を直接規定している: PRAGMA(WAL/foreign_keys)、migrations テーブルによる適用履歴管理、「新規 DB は最新スキーマ+全マイグレーション既適用マーク」という原典準拠のマイグレーション意味論。FK の CASCADE/SET NULL も仕様の一部(REQ-028)。

## 決定
- ドライバ: **Microsoft.Data.Sqlite 10.0.9**
- マッピング: **Dapper 2.1.79**(クエリ結果→レコード型)
- スキーマ: **手書き DDL** + 自作マイグレーションランナー(REQ-004 の意味論をそのまま実装)
- リポジトリパターン(インターフェースは Core、実装は Infrastructure)
- 接続戦略: 単一共有接続 + 非同期ミューテックスでシリアル化(WAL。V1 の規模では十分。G2' AUDIT-103 の解消)

## 却下した代替案
- **EF Core**: マイグレーション履歴(__EFMigrationsHistory)の意味論が REQ-004 の契約(migrations テーブル、新規 DB 全マーク)と競合する。スキーマ・SQL が仕様で固定されているため、抽象化の利益よりモデル⇔仕様の二重管理コストが大きい
- **生 ADO.NET のみ**: マッピングの手書きは冗長でずるの温床。Dapper は十分薄い

## 影響
- DDL・マイグレーション・SQL は M-BOM の製造単位(M-DB-007)として明示的に受入される(L2: スキーマ同値検査)
- PRAGMA は接続確立ごとに明示発行する(既定値に依存しない — K-SQLITE)
